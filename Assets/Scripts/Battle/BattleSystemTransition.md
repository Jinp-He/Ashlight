# Battle System Transition Guide

## 目标
将当前 `README.md` 中定义的 `Timeline Battle System`，平滑迁移为 `NewBattleSystemReadme.md` 中定义的 **离散行动条 ATB Battle System**，并确保以下核心特性不被破坏：

- 纯数据驱动
- 逻辑与 UI 分离
- 确定性计算
- 深拷贝支持
- 真实战斗与预测系统规则一致

---

## 迁移总策略
本次迁移不能采用一次性整体替换的方式，而应采用分层迁移策略：

1. 保留当前 Timeline 系统可运行
2. 在状态层补齐新的离散行动条所需数据结构
3. 在引擎层逐步建立新的离散 ATB 解算链路
4. 将卡牌、预测、Buff、AI 等模块逐步切换到新链路
5. 最后清理 Timeline 专属结构和旧入口

迁移的核心原则是：

- 先统一状态，再替换逻辑
- 先建立新链路，再移除旧链路
- 预测系统必须与真实执行共用同一套规则
- 过载机制必须进入核心状态流，而不能只做局部补丁
- 行动节奏必须以**离散段位**而非连续百分比作为主要规则语义

---

## 现状与目标的本质差异

### 当前系统：Timeline Battle System
当前系统基于离散时间格推进：

- 每个单位拥有 `TimelineTrack`
- 每张卡牌会转成 `TimelineBlock`
- 行动按时间格逐格解算
- 同一格内通过优先级决定执行顺序
- `TimelineResolver` 负责解算时间推进

### 目标系统：Discrete ATB Battle System
目标系统基于 **离散行动条段位** 推进：

- 每个单位拥有 `ActionBarState`
- 行动条被划分为若干固定段位，而不是连续百分比
- 单位根据 `Speed` 在行动条段位上推进
- 当单位到达终点段位时，获得一个完整回合
- 回合中完成能量刷新、抽牌、出牌、结算和重启
- 支持 `Overload` 机制，以未来节奏换取当前额外出牌能力
- 过载的主要代价表现为：**下次起始段位后退若干段**

### 迁移本质
这次迁移的本质不是调整局部算法，而是将战斗系统从：

`离散时间格驱动模型`

迁移为：

`离散行动条段位驱动模型`

因此需要重新审视以下内容：

- 行动顺序来源
- 卡牌行为入口
- 状态更新时间点
- 时间类效果语义
- 预测系统模拟链路
- 行动条段位与区间的表达方式

---

## 模块对应关系

| 旧系统模块 | 新系统模块 | 迁移说明 |
|---|---|---|
| `TimelineTrack` | `ActionBarState` | 从固定时间格结构变为离散行动条段位结构 |
| `TimelineBlock` | 回合内命令序列 | 不再作为核心运行单元保留 |
| `PhaseEnum` | 回合流程阶段 | 从块阶段转为回合阶段概念 |
| `TimelineResolver` | `ActionBarResolver` + `TurnResolver` | 拆分为节奏判定与回合执行 |
| `CardToTimelineConverter` | `CardPlayResolver` | 卡牌不再转成时间块，而是直接解算命令 |
| `TimeShiftCommand` | `ActionBarShiftCommand` | 语义从格位移动改为行动条段位推进/后退 |
| 无核心过载结构 | `OverloadState` + `OverloadCommand` | 新系统新增核心机制 |
| 旧 `BattlePredictor` | 新 `BattlePredictor` | 必须与真实离散 ATB 规则完全一致 |

---

## 迁移步骤

### 第一步：统一状态层
优先处理所有核心状态数据结构，使战斗状态快照能够容纳新系统运行所需的最小集合数据。

这一阶段需要重点统一：

- 单位基础属性
- 当前生命与护甲
- Buff 列表
- 当前手牌与能量
- 行动条当前段位
- 回合结束后的重启段位
- 过载负债

目标是让 `BattleStateSnapshot` 和 `UnitState` 成为新旧系统可共用的基础状态载体。

注意事项：

- 不要让旧系统状态和新系统状态并列为两个主来源
- 所有后续解算器都应围绕统一状态快照运行
- 行动节奏相关状态必须优先表达为“段位”而不是“浮点进度”

---

### 第二步：补齐离散 ATB 专属数据结构
在现有核心数据层中引入或补齐以下目标结构：

- `ActionBarState`
- `OverloadState`
- `CurrentSegment`
- `RestartSegment`
- `OverloadDebt`
- 当前回合资源字段
- 可选的 `ActionBarZone` / `SegmentBand` 概念（用于临界区、过载区等）

这一阶段的目标不是立即启用离散 ATB 主链路，而是为后续解算器建立正确的数据基础。

注意事项：

- `TimelineTrack` 在过渡期可以保留，但不能继续作为长期唯一主节奏来源
- 新字段应逐步成为后续功能的默认依赖
- 不建议新系统继续依赖连续百分比或浮点进度作为主要对外语义

---

### 第三步：重建行动顺序解算
实现 `ActionBarResolver`，替换旧系统中基于时间格推进的节奏判断方式。

需要承担的职责包括：

- 根据 `Speed` 推进行动条段位
- 确定下一位获得行动权的单位
- 处理 `OverloadDebt` 对未来节奏的影响
- 在回合结束后计算新的 `RestartSegment`
- 处理单位处于不同区段时的状态标签（如准备区、临界区、过载区）

这一阶段是对旧 `TimelineResolver` 节奏职责的拆解与替换。

注意事项：

- 旧系统的推进单位是时间格
- 新系统的推进单位是行动条段位和行动权
- 不能仅通过改名复用旧解算器
- 推进规则必须优先支持“进入某段”“离开某段”“到达终点段”等离散事件

---

### 第四步：建立完整回合解算器
实现 `TurnResolver`，让单位在获得行动权后可以完成一个完整回合。

一个标准回合至少应包括：

1. 结算回合开始效果
2. 刷新能量
3. 抽牌
4. 出牌
5. 在能量不足时允许过载
6. 执行所有命令
7. 更新 Buff / Debuff
8. 写回新的行动条起跑段位

这一层是整个离散 ATB 系统的核心执行闭环。

注意事项：

- 旧系统偏重时间格触发
- 新系统偏重单位回合闭环
- 如果没有独立的 `TurnResolver`，卡牌和预测系统就无法真正脱离时间轴模型
- 过载造成的节奏损失，应优先以“后退若干段”表达，而非连续时间惩罚

---

### 第五步：迁移卡牌执行入口
将原本 `CardToTimelineConverter` 的职责迁移到 `CardPlayResolver`。

旧系统流程：

- 卡牌效果先转换为 `TimelineBlock`
- 插入时间轴
- 到对应格时执行

新系统流程：

- 卡牌效果直接转换为 `Command`
- 在回合内按规则结算
- 若涉及节奏变化，则通过 `ActionBarShiftCommand` 处理段位变化

这一阶段建议尽量保留 `Effect -> Command` 的总体设计，减少配置层重写范围。

注意事项：

- 不应再围绕“插入未来某格”来设计主要入口
- 卡牌的节奏效果应以“推进若干段”“后退若干段”“进入临界区”“对某区段生效”等形式表达

---

### 第六步：重定义时间类效果
当前系统中的 `TimeShiftCommand` 依赖时间格偏移语义，迁移后需要改写为离散 ATB 语义。

新系统中应重新定义为：

- 推进或拉回行动条若干段位
- 影响单位下次行动的到达顺序
- 必要时影响回合后起跑段位
- 与过载节奏影响保持一致
- 支持基于区段的判定与奖励

注意事项：

- 不能将“推后几格”直接等价替换为“减慢一点”
- 需要统一数值意义，否则卡牌节奏表现会失真
- 建议统一成“前进/后退 N 段”的设计语言
- 若存在区域打击、同步打击等子玩法，应优先基于“同段”“相邻段”“临界段”等语义实现

---

### 第七步：接入过载机制
`Overload` 是新系统的核心特色，必须作为独立阶段接入。

需要明确：

- 什么时候允许过载
- 过载消耗或追加的代价
- `OverloadDebt` 如何累计
- 负债如何影响下次行动节奏
- 是否附加脆弱性或其他风险
- 预测系统如何展示过载结果
- 过载是否会导致下次起始段位直接退入过载区

这一阶段完成后，离散 ATB 才真正具备目标文档中的节奏特征。

注意事项：

- 过载的节奏惩罚应优先离散化
- 例如：轻度过载后退 1 段，中度后退 2 段，重度退入过载区
- 若系统需要承载“过载补偿”玩法，则必须允许某些效果在过载状态或过载区中获得强化

---

### 第八步：重做结算时机
当前 Timeline 文档中的结算规则偏向“每个时间格解算完成后处理”。迁移到离散 ATB 后，必须重新定义结算边界。

需要重新区分：

- 命令执行后立即处理的内容
- 回合结束时处理的内容
- 行动条段位推进后统一处理的内容
- 单位进入特定区段时处理的内容
- 单位死亡时立即中断或清理的内容

这一阶段会直接影响：

- Buff 更新时机
- 护甲清空规则
- 死亡判定时机
- 战斗结束判定逻辑
- 区段触发类效果的稳定性

---

### 第九步：迁移预测系统
新 `BattlePredictor` 必须建立在离散 ATB 的真实解算链路之上，而不是继续调用旧 Timeline 规则。

预测流程应为：

1. 克隆当前战斗状态
2. 在克隆状态上执行同一套离散 ATB 规则
3. 输出预测结果

预测结果建议至少覆盖：

- `HpChangeMap`
- `BlockChangeMap`
- `BuffChangeMap`
- `WillKillTargets`
- `ExpectedDamage`
- `ExpectedOverloadDebt`
- `ExpectedSegmentShift`
- `ExpectedRestartSegment`
- 可供 UI 使用的摘要信息

核心要求：

- 预测不能修改真实状态
- 预测结果必须与真实执行一致

注意事项：

- 预测系统不应继续输出旧式“时间格位置变化”作为主要信息
- 应优先输出“目标后退 1 段”“将进入临界区”“下回合从第 2 段起步”等离散结果

---

### 第十步：迁移外围调用方
当核心链路稳定后，再逐步切换外部依赖模块。

优先迁移对象包括：

- 卡牌拖动预览
- 目标切换预览
- 敌方伤害预测
- AI 局部评估
- 战斗 UI 展示

建议做法：

- 对外保留统一接口
- 外围系统不直接依赖底层 Timeline 结构
- 内部逐步将实现切换到离散 ATB 链路
- UI 层优先表达“区段、临界区、过载区、后退若干段”等概念

---

### 第十一步：清理 Timeline 旧模块
只有在以下条件全部满足后，才应开始移除旧 Timeline 模块：

- 新离散 ATB 真实战斗已完整跑通
- 新预测系统与真实执行一致
- 卡牌系统已切换到新入口
- Buff 与结算时机稳定
- UI 与 AI 不再依赖 Timeline 专属结构

满足条件后，可逐步废弃或移除：

- `TimelineTrack`
- `TimelineBlock`
- `PhaseEnum`
- `TimelineResolver`
- `CardToTimelineConverter`
- 旧语义下的 `TimeShiftCommand`

---

## 迁移中的关键约束

### 1. 不要长期并存两套主逻辑
过渡期允许兼容，但不能让 Timeline 和离散 ATB 长期作为正式主链路同时维护，否则会导致：

- 状态重复
- 逻辑分叉
- 预测不一致
- UI 绑定来源混乱

### 2. 优先保留 Effect 层
从文档结构看，旧系统与新系统都保持 `Effect -> Command` 思路，因此迁移时应优先保留 Effect 层，重点重写命令语义和解算入口。

### 3. 预测系统必须与真实系统共用规则
这是迁移正确性的核心要求之一。只要预测和真实战斗不走同一套规则，系统就会出现显示结果与实际结果不一致的问题。

### 4. 过载必须进入核心状态流
`Overload` 不能只停留在单张卡牌逻辑或显示层，它必须影响：

- 回合内出牌能力
- 行动条段位推进
- 回合结束重启段位
- 预测结果输出

### 5. 规则语义必须离散化
既然新系统决定采用离散行动条表达，则以下内容都不应继续以连续百分比作为核心设计语言：

- 时间类效果
- 过载惩罚
- 敌人节奏干扰
- UI 预览结果
- 子玩法判定条件

建议统一使用：

- 前进 / 后退若干段
- 位于某区段
- 进入 / 离开临界区
- 与目标处于同段 / 相邻段

---

## 建议验收标准
迁移完成后，至少应满足以下标准：

- 相同输入的战斗状态产生完全一致的输出
- `BattlePredictor` 的结果与真实执行结果一致
- 单位速度能够稳定影响行动节奏
- 回合开始、回合结束、Buff 更新时机定义清晰
- 过载可改变当前回合操作能力，并真实影响后续节奏
- 行动条节奏与过载惩罚可用离散段位清晰表达
- 旧 Timeline 模块移除后，系统仍完整可运行

---

## 推荐实施顺序
如果进入实际开发，推荐按以下顺序推进：

1. 调整 `UnitState` 与 `BattleStateSnapshot`
2. 补齐 `ActionBarState` 与 `OverloadState`
3. 明确行动条段位划分与区段语义
4. 实现 `ActionBarResolver`
5. 实现 `TurnResolver`
6. 迁移到 `CardPlayResolver`
7. 重写时间类效果为 `ActionBarShiftCommand`
8. 接入 `OverloadCommand`
9. 迁移 `BattlePredictor`
10. 清理 Timeline 旧模块

---

## 结论
这次迁移不是局部替换某个解算器，而是一次核心战斗模型的重构。正确顺序应当始终是：

`状态层统一 -> 节奏解算替换 -> 回合解算建立 -> 卡牌与命令迁移 -> 预测系统迁移 -> 外围系统切换 -> 旧模块清理`

在新的目标系统中，这条链路应建立在：

`离散行动条段位 + 单位回合闭环 + 过载后退段位`

这套统一语义之上。

只有按照这个顺序推进，才能在保证系统可运行的前提下，将当前 Timeline 战斗系统稳定迁移到新的离散 ATB 战斗系统。
